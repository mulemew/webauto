# 生产环境验证清单

本批改动的逐项验证步骤。**本地没有 node / python,这批全部只过了 CI 的 TypeScript typecheck,`server.py` 连语法检查都没做过** —— 下面标 ⚠️ 的项目是完全没跑过的路径,请优先测。

---

## A. 转盘 / 点击

### A1. text 点击不再"报成功却没点上" ⚠️
**背景**:老代码给元素打临时属性 `data-wa-textclick` 再按属性点;Vue 页面重渲染会抹掉它 → 点了个空但报成功。现在改成推导元素自身的稳定选择器直接点。

1. minestrator 转盘任务,步骤用 **text 类型**、填 `Spin the wheel`
2. 跑一次
3. **期望**:步骤消息形如 `Clicked element matching text "Spin the wheel" [real click] — page reacted (N DOM changes)`,N 应该很大(几十~上千)
4. **失败信号**:`NO page reaction (0 DOM changes)` → 说明还是没点上,把消息发我

> 备选:选择器类型改 **CSS**、填 `button.wheel-cta`(已验证可用)。

## B. wispbyte GitHub 登录

### B1. 图标按钮能被找到 ⚠️
**背景**:wispbyte 的 GitHub 按钮是纯图标 `<div class="continue-via"><i class="fa-brands fa-github">`,无文字/href/onclick,老代码三种匹配全都够不着。

1. 跑 wispbyte 的 GitHub OAuth 登录任务
2. **期望**:日志出现 `Clicked GitHub OAuth trigger`,并跳到 github.com
3. **失败信号**:仍报 `Already authenticated` 或 `Could not find "Sign in with GitHub" button`

### B2. 不再误报"已登录"
1. 用一个**未登录**的 wispbyte 任务
2. **期望**:如果按钮没找到,报 **`Could not find "Sign in with GitHub" button on login page`**(诚实失败)
3. **失败信号**:报 `Already authenticated. URL: https://wispbyte.com/client` 却其实没登录(旧 bug)

## E. WARP 换 IP 打 reCAPTCHA ⚠️(全新,最需要实测)

### E1. WARP 不再需要预生成配置
1. 任务代理类型选 **warp**,**不设** `WARP_CONFIG_PATH`
2. 跑任务
3. **期望**:能正常起代理(Node 内直接注册 WARP 身份)
4. **失败信号**:`WARP registration failed: HTTP xxx` 或 `WARP registration returned an unexpected payload` → 把日志发我(Cloudflare 接口返回结构可能要调)

### E2. 每任务配置换 IP 次数(#1)
1. 编辑任务 → 代理类型选 **warp**
2. **期望**:下方出现 **"换 IP 重试次数"** 输入框(选别的类型时不出现)
3. 填 3 → 保存 → 重新打开,值还在

### E3. 被 block 时真的会换 IP
1. 跑一个会遇到 Google reCAPTCHA 的 warp 任务
2. **期望**日志按序出现:
   - `reCAPTCHA audio blocked for this exit IP — rotating proxy IP and retrying`
   - `WARP identity rotated (same local SOCKS port)`
   - 重复直到成功或用完次数
3. 成功时步骤消息带 **`(after N IP rotation(s))`**
4. 失败时消息带 **`still blocked after N IP rotation(s)`**
5. **关键**:换 IP 期间**浏览器不应重启**,页面/表单状态保留

## F. Cloudflare 全页验证

### F1. 不再卡十几分钟(#4)
1. 跑一个登录页在 CF 全页验证后面的任务
2. **期望**:CF 清理**最多约 90 秒**就出结果(成功或失败),不再阻塞十几分钟
3. 超预算时日志:`Cloudflare interstitial clear exceeded its time budget — aborting`
4. 可用 `CF_CLEAR_BUDGET_MS` 调整

### F2. 全页验证的方框会被点 ⚠️
1. 同上任务
2. **期望**:日志出现 `coordinate fallback (interstitial): checkbox at (x,y)`,且验证通过后返回 `method: "coord-interstitial"`
3. **失败信号**:仍是 `{"solved": false, "method": "none"}` → 方框还是没点上,发我日志

## H. 代理预检(#12)

### H1. 代理挂了秒失败
1. 把任务代理故意改成一个**不通的地址**
2. 跑任务
3. **期望**:**几秒内**失败,消息 `Proxy precheck failed: target is unreachable through the configured proxy — the proxy is down or the site is blocking it`
4. **失败信号**:还是慢慢跑完整个浏览器流程才失败

### H2. 正常代理不受影响
1. 代理正常的任务
2. **期望**:日志有 `Proxy precheck: HTTP 200 via xxx`,任务照常继续
3. **注意**:目标返回 403/503(CF 拦非浏览器请求)**不算失败**,会继续跑

## J. 僵尸进程回收(#10)⚠️

1. 部署前先看基线:`docker exec <app容器> ps aux | grep -E "sing-box|Xvfb" | wc -l`
2. 跑几个带代理的任务,然后 `docker compose restart app`
3. **期望**:重启后 sing-box / Xvfb **不累积**(数量回到基线,不是每次 +N)
4. 日志应见:`Shutting down — helper processes reaped`(带 `killedHelpers` 数)
5. **重点**:以前每次重启都会留下孤儿 Xvfb(detached 且没人回收),现在应该没有了
